# 調査ミス（誤字・圧縮/改変引用・同期漏れ）の根本原因考察（2026-07-22）

対象: note 全件検証で検出した25件＋前回検出分（exactDays 19件・ages.note 矛盾7件）。
再発防止のためのプロセス改訂案を含む。git 履歴で混入時期を裏取り済み。

## 1. 事実の整理

- エラー率は引用ユニット 6,504 中 25 件 ≒ 0.4%。壊滅的ではないが**特定の経路に集中**している
- git -S 追跡の結果、混入時期は一つのフェーズに偏らない:
  - 弋樂/燈王 → 2026-07-17 グループ3・4 調査パス（5ee615f）
  - 蕭謐/絶興 → 2026-07-21 フェーズB 出典差し替え（f2103a3・6addef1）
  - 曲賜 → 初期データセット（acf7bca）
  = **「モデルが文字を書き写す局面」があるたびに一定率で発生**しており、特定バッチの品質事故ではない

## 2. 直接メカニズム（5類型）

### (a) モデル書写経路（LLM が原文を「打ち直す」）
検出誤字は字形・字義の近似置換（蕭謐↔諶、惼↔惧、賛↔贇、絶興↔紹興、曲賜↔曲赦、駆↔馳）と
稀字での崩れ（燈王/袲之＝文字化け様）。これは LLM が原文をコピーせず**文脈から再生成**するときの
典型的な変異パターン。ツール出力の貼り付けなら起きない。

### (b) モデルによる字体変換の暗黙実施
コーパスは簡体（daizhigev20）・繁体/原文HTML（china-history）混在、note は日本語文脈の
繁体・新字体混じり。**変換をモデルが頭の中でやる**ため、猎→獵のつもりが樂になる類の
lossy 変換が混入。検証時も新字体→簡体の未変換で notfound の過半が擬陽性化した（同根の問題）。

### (c) 記憶引用・再構成（見つからないときの補完）
- chen-houzhu「毎日與子弟飲酒一石」: 陳書に無く南史/通鑑「日飲一石」の改変＝**有名な逸話を
  記憶から再構成**し、期待される出典（調査規約上の第一情報源＝正史本紀）に帰属させた
- sui-yanghao: 引用実在だが出典は通鑑（note は隋書列伝と主張）＝**期待出典バイアス**
- dongjin-yuandi「帝忿憂成疾」: コーパスに無い引用風漢文＝二次資料の要約の引用化
- gongsun-shu「明日死」（原文「其夜死」）: 要約時の事実改変
規約が「正史原典の直接引用」を要求する一方、**見つからなかったときの明示的な逃げ道
（「該当文なし」と書く規約）はあったが、逸話が有名なほど再構成の誘惑が勝った**。

### (d) 非原子的な多段更新（同期漏れ）
12項目×複数フェーズ×並行セッションで同一レコードを別々の時期に更新した結果:
- tangmo-shisiming: endDate は訂正、ages.note の旧日付は残存
- ages.note「null とした」×7件: 後続パスが値を埋め、note を更新せず
- liao-jingzong: conversion が「1日ずれ・不一致」と**発見まで記録しながら**解決タスク化されず放置
「note と値を同じタイミングで更新」の規約はあったが、**機械検証が無く規約頼み**だった。

### (e) ツーリングの暗黙前提
- exactDays 19件: ユリウス暦ラベルに datetime（グレゴリオ算術）を適用。**暦系という前提が
  コードにもドキュメントにも明示されず**、閏世紀日をまたぐ19件だけで静かに顕在化
- liang-xiaoyuanming: conversion 文の fromLunar 引数が誤記＝**実際に実行した換算とは別に、
  後から人手（モデル）で記述を書いた**ため、実行と記録が乖離

## 3. 根本原因（why の集約)

1. **「文字がモデルを通過する」経路が検証されていなかった**。構造 CI（count 整合・スキーマ・
   日付形式）は強いが、note⇔値⇔原文の突合はゼロだった（今回初めて実施し、初回で25件検出）
2. **引用が検証可能な形式で保存されていない**。出典はあるが「コーパスのどのファイルの何行目か」が
   無いため、機械照合が今回のような大掛かりな突合になる
3. **トークン効率化の圧力**が要約・圧縮引用を助長（Workflow の return 要約化方針と同時期）
4. **発見済み不一致の追跡機構が無い**（conversion 内の「ずれ」記述が terminal state になれる）

## 4. 再発防止策（今後の調査プロセス規約）

### 調査エージェントの規約（RESEARCH_PROCESS.md へ追記する案）
1. **引用手打ち禁止**: quote/note 内の原文引用は、必ず grep/Read のツール出力からコピーする。
   引用には `_corpus_cache/<id>.txt` の行番号（またはコーパスファイルパス＋行番号）を併記する
2. **字体変換禁止**: 引用はコーパスの字体のまま保存する（簡体コーパスなら簡体のまま）。
   変換が必要なら opencc 等の機械変換のみ
3. **見つからない場合の明示**: 原文が確認できない場合は引用の体裁を使わず
   「該当原文は確認できず（○○による）」と書く。二次資料由来は出典名を必須にする
4. **換算の記録はスクリプト生成**: conversion 文は実行した sxtwl 呼び出しのログから機械生成する
   （実行と記録の乖離を構造的に不可能にする）
5. **不一致は未解決として登録**: 調査中に発見した「ずれ・不一致」は note に書いて終わりにせず、
   KNOWN_ISSUES（validate_emperors.py）か task.md に登録して解決まで追跡する

### 機械ゲート（validate_emperors.py / CI 恒久化の案）
6. **引用実在チェックの定常化**: 今回の verify_quotes（新字体→簡体変換の改善込み）を
   ブロック完了ゲートに組み込む。増分（新規・変更レコードのみ）なら常時実行可能
7. **暦換算リプレイの定常化**: verify_calendar（fromLunar リプレイ＋exactDays JDN 再計算）を
   CI に追加。暦系の前提（1582-10-15 改暦・ユリウス JDN 差）をコードとして固定
8. **note⇔値突合**: 「null とした」型矛盾（N9）・note 内日付主張 vs 値（B4）・
   conversion 内の不一致語（「ずれ」「不一致」「合わない」）検出を警告化

### 検証側の教訓（AI_RESEARCH_LESSONS.md へ追記する案）
- 照合正規化は opencc だけでは不足。日本語新字体→簡体の変換表が必須
- daizhigev20 旧五代史.txt は部分版（後晋以降は旧五代史四库.txt）。明史巻四に洪武31年末の欠落
- 全A判定・全シロ報告はアンカリングを疑い、陽性・陰性の両側を抜き打ち再検証する
