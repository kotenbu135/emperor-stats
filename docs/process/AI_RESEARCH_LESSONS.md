# 史書をAIで調査する方法の知見集

中国皇帝364人×12項目のデータセット構築（2026-07-15〜18、調査エージェント延べ数百体）を通じて得られた、**AI（LLMエージェント）で史書を大規模調査するための一般化可能な知見**をまとめる。

- 「何をするか」の作業手順は [RESEARCH_PROCESS.md](RESEARCH_PROCESS.md)、史料ごとの具体的マッピングは [CORPUS_NOTES.md](CORPUS_NOTES.md)、絶対的な制約は [CONSTRAINTS.md](CONSTRAINTS.md) を参照。本書は「なぜその方法に落ち着いたか・何が失敗したか」の記録であり、同種のプロジェクト（史料ベースの悉皆調査全般）を再び行う際の設計指針として読む。

## 1. 情報源の信頼性 — AIの要約を判定根拠にしない

**LLMは「もっともらしい要約」を作るのが得意なため、要約を判定根拠にすると誤りが混入しても気づけない。** 判定は必ず原典の該当箇所（本紀・列伝の原文）まで降りて行う。

- **WebSearch のスニペットだけで判定しない。** スニペットが史書原文を直接引用しているか、学術的信頼性のあるソース（維基百科・百度百科の原典引用部分、研究者のページ等）かを確認する。個人ブログ・まとめサイトのみが根拠の場合は confidence を下げるか原典に当たり直す（更始帝の死因調査でユーザー指摘を受けて確立した方針）。
- **AI翻訳サイト・解説サイトは「目次」として使う。** 二十四史のAI和訳サイト（aki-asahi.com）は該当人物・巻の当たりをつけるのに有用だが、AI翻訳自体を確定根拠にはしない。必ず原文で裏取りする。
- **史書自体のバイアスを織り込む。** 易姓革命後に編纂された正史は前王朝最後の皇帝を実際以上に悪く描く傾向が知られている。原典の記述をベースにしつつ、現代の学術的通説と食い違う場合は「諸説あり」として両論併記する。
- **「不明」を正式な調査結果として扱う。** 無理に一つの答えを決めず「不詳」「諸説あり」を用意する。さらに悉皆調査では**「調査済みだが不明」（空値＋note）と「未調査」（フィールド省略）を構造上区別する**ことが重要 — この区別がないと「調査漏れ」と「調べたが史料に記載がない」が混ざり、進捗管理も完了検証もできなくなる。五胡十六国・南北朝の傀儡皇帝や極短命の簒奪者では「生年不明が正しい調査結果」であることが多く、76人中44人が「調査済みだが不明」で確定したブロックもあった。
- **候補人物リスト自体の抜け漏れも疑う。** 「364人全員完了」は候補リスト内の悉皆性であり、候補リスト自体が完全である保証にはならない。実際、通史年表の設計検討で在位年カバレッジを機械集計（`reigns[].startYear`〜`endYear`の年ごとの在位人数）したところ905-906年が皇帝不在期間として現れ、唐哀帝（李柷）の収録漏れが判明した（唐は殤帝・李裕等の短命帝まで収録済みだったのに哀帝だけ欠落。2026-07-20発見）。この種の「連続性の破れ」を機械的に検出する手法（年ごとの在位人数カバレッジ集計、王朝ブロックの先頭/末尾皇帝の在位終了年と次王朝の開始年の突き合わせ等）は、候補リストの抜け漏れを見つける有効な監査手段になる。
- **旧暦→新暦換算（`sxtwl`）には日単位で1日程度のブレが生じ得る。** 同一人物・近接する日付でも、既存レコードとの突き合わせで新規換算が1日ズレるケースがあった（唐哀帝の崩御日「天祐五年二月二十一日」を`sxtwl`換算した日と、`資治通鑑`の干支表記「癸亥」から逆算した日が1日食い違った）。原典に直接の日次記載（「◯月◯日」）がある場合はそれを優先して換算し、干支のみのクロスチェックで1日程度の差が出ても無理に一致させようとせず、`verification.notes`に差異を記録した上で直接記載側を採用する。近接する既存レコード（例: 別人物だが同時期の即位/崩御日）との整合確認は、換算ミスの発見に有効。

## 2. 史料はローカルに置く — 外部サイトを叩かない

**数百人分の調査で外部サイトを都度取得するのは、効率面でも礼儀面でも破綻する。** 整理済みのオープンデータを1回だけ `git clone` してローカル参照するのが正解だった。

- 当初 ctext.org を都度 WebFetch していたが、Bot判定・アクセス制限のリスクがあり2026-07-16に全面廃止。GitHub公開のコーパス2種（china-history＝二十四史HTML、daizhigev20＝殆知閣テキスト）をローカルに取得し、以後の原文確認は「ローカルコーパス → だめなら維基文庫」の2段構えにした。
- **外部APIへの一括リクエストは「小規模検証 → 想定件数を提示して明示的許可 → 本実行」を必ず踏む。** 肖像画調査で Wikimedia API に364件を並列で投げて 429 と乱用警告を受けた失敗がある。429 が出たらリトライ・バックオフで粘らず即停止して方針自体を相談する。可能なら公式の一括ダンプ（オフライン処理）を優先する。

ローカルコーパスには固有の罠がある（詳細は [CORPUS_NOTES.md](CORPUS_NOTES.md)）。一般化すると：

- **ファイル名・ラベルを信用しない。** 列伝ディレクトリの巻数が絶対巻数でなく相対番号の書がある（宋史・金史・明史）。「原文」ラベルなのに中身が白話訳（現代語訳）というデータ不備もある（晋書の一巻で実例）。新しい書に着手する際は必ず1〜2件を原文の「卷○○」見出しと照合し、文体（文言文か現代語調か）を目視確認してからエージェントに配る。
- **OCR/転記由来の誤字がありうる**前提で、直接引用は文脈と照合する。避諱表記・PUA文字・人物記事の巻をまたぐ分散（朱友珪など）も個別に発生する。
- **発見した罠はブロックごとのメモに散在させず、1つのチェックリストに集約する。** 同種の罠を次のブロックでゼロから再発見するコストが実際に発生していた。

### 前処理の3段活用（コンテキスト効率の土台）

1. **行番号インデックス**: 章分割のない大容量ファイル（清史稿.txt 15.6MB等）は、本紀見出しを `grep -n` で洗い出して皇帝ごとの行範囲表を作り、エージェントには該当行範囲だけを Read の offset/limit で渡す。原文自体は全文読めるため精度は落ちない。
2. **人物ごとの原文キャッシュ（`_corpus_cache/<id>.txt`）**: 調査項目が複数グループに分かれていると、同じ人物の同じ本紀を最大4回読み直す無駄が発生する。コーパスから人物ごとの原文を**機械的に抽出するだけ**（判定は一切しない）のスクリプトで事前にキャッシュ化する。当初は「もう読み直さないブロックには不要」と選別していたが、「調査項目は後から増える」という理由で全ブロック標準運用に転換した — **悉皆調査では『この史料はもう読まない』という予測は当てにならない**。
3. **インデックス自体を実地検証する**: 行番号表の1エントリが誤っていた実例がある（清史稿の宣統本紀の終了行が志の途中を指していた）。表を作ったら代表箇所を `grep -n` で再検証してから使う。

## 3. 判定の設計 — 何をAIに任せ、何を機械化するか

**原則: 史実の判定はAIエージェントの個別調査、構造検証・転記・集計はスクリプト。** この線引きが最重要で、逆転させると両方失敗する。

- **判定のスクリプト化（データの自動生成）は禁止。** このプロジェクトでは過去2回失敗している。人物ごとに1エージェントが原文を読んで判定する方式が最終的に364人×12項目を通してエラー率ほぼゼロで完走した。
- **数え方の基準は着手前に文書化してユーザー承認を得る。** 「親征1回とは何か」「反乱と統一戦争の境界はどこか」のような基準は、調査中に必ず境界事例が出る。事前基準＋境界事例の都度ルール化（例: 簒奪前の挙兵は即位経路の領域として反乱に数えない、対立政権間の戦争は簒奪側と被簒奪側で非対称に計上する、宋遼西夏金は相互に外国関係として扱う）を基準文書に追記し続けることで、後続ブロックの判定が安定する。
- **基準を変えたら過去分を遡及監査する。** 被反乱回数の算入基準を途中で訂正した際は、訂正前に調査済みだった35名を監査して整合させた。
- **調査単位は項目の性質で選ぶ。** 
  - 皇帝個人の行為（改元・親征等）→ 人物ごと1エージェントの本紀通読。
  - 王朝単位の離散イベント（遷都）→ 王朝単位でイベント年表を先に確定し、在位期間へスクリプトで逆引き割当（364エージェント→30エージェントに削減）。ただし逆引きには同名別政権の衝突・皇帝号採用前のイベント・対立皇帝の並立など機械照合で解けないケースが残るため、「自動マッチ → 曖昧なものは note を読んで手動オーバーライド → 構造的グレーゾーンは既存の例外規定を類推適用」の3段構えにする。
  - 既存調査の副産物が使える項目（年齢）→ 新規の原典探索より先に、既に支払い済みの調査コスト（死因・即位経路調査の note・出典ページ）の再確認を起点にする。
- **同じ原文を読む複数項目は1エージェントに合算する（合算パス）。** グループ3（親征等）とグループ5（年齢）を合算した結果、7ブロック・188名を通して精度劣化なし・機械検証エラーほぼゼロで完走した。ただし導入時は**明示的なロールバック条件**（見落とし増加・検証エラー率悪化・削減効果僅少）を決めた試験導入とし、ブロックごとに条件該当の有無を記録して継続判断した。
- **旧暦→太陽暦換算は「日番号と干支の二重記載」「在位中崩御なら reigns.endDate」との突合で必ず裏取りする。** ages 日付の ISO 正規化（2026-07-21、62件）では、干支と日番号の両方が原文にある日付は sxtwl 換算で両者の一致を確認し、在位中崩御の 17 件はフェーズB検証済みの endDate と完全一致することを機械 assert した。この突合で、原文転記時の干支誤記 2 件（後唐荘宗の生日「癸亥」→sxtwl では 22 日=癸酉、崩御「丁丑」→四月朔は丁亥）を検出できた。また **4 世紀（東晋期）の日付は sxtwl（定朔ベース）と当時の官暦（平朔）の朔差で干支が旧暦月内に見つからないことがある**（姚興の「二月丁未」・石遵の「十一月丙辰」で発生）。この場合は日精度をでっち上げず月精度に落とし、旧暦月の太陽暦多数月を採用する。旧暦月だけ判明している日付の月精度換算も同じく多数月方式（例: 旧暦九月→大半が重なる 10 月）で統一する。 出典QAフェーズBブロック1で `reigns[].endDate` を訂正した際、同じ崩御日を持つ `ages.deathDate`（4件）と note 内の日付引用（2件）の同期を取りこぼし、公開サイトの JSON-LD に旧値が出たままライブ検証まで気づけなかった。訂正対象のフィールド名だけ直すのではなく、旧値の文字列でレコード全体（および表記規約が同じはずの隣接フィールド）を grep して残存参照を列挙するのを訂正手順の定型に含める。

## 4. 並列エージェント運用 — 構造化と検証

数十体の並列エージェントから返る結果を人手で扱うと、転記ミスと目視コストで破綻する。**「LLMの出力をLLMが書き写す」工程をゼロにする**のが設計目標。

- **構造化出力（schema強制）を必ず使う。** 自由文JSONを手で heredoc に書き写すのは、漢文引用を含む長文の再入力であり改変・脱字リスクがある（実際に一度やってしまい、以後禁止した）。
- **id はプロンプトでピン留めする。** エージェントは人物名から勝手にIDスラッグを自己生成しがち（あるブロックでは35名中20件超が不一致）。「返すJSONの `id` は指定した `<id>` を一字一句そのまま使うこと。独自生成禁止」という文言と、プロンプト本文への id 直接埋め込みで、以後のブロックは不一致ほぼ0件になった。
- **マージスクリプトに機械検証を内蔵する。** `count == events.length` の一致、スキーマ外キーのホワイトリスト除去、id照合をスクリプトが自動チェックし、不一致ならエラー終了。毎回の目視チェックを廃止できた上、実際に count 更新忘れ等の軽微ミスを複数ブロックで機械検出した。
- **マージは journal（エージェントの構造化出力の生記録）から直接組み立てる。** メイン会話に表示された要約からではなく、実行記録ファイルをスクリプトで読む。表示は「人物ごとのカウント表＋エラー有無」の要約のみで、保存データの粒度は落とさない。
- **完了宣言の前に必ず悉皆実測する。** チェックリスト上は「364人完了」でも、jq でフィールド保有数を実測したら355人しかいなかった事例がある。原因は、合算パスで後からデータセットに追加されたブロックが、既に完了扱いだった別グループのチェックリストに遡及追加されていなかったこと。**チェックリストは自己申告であり、完了判定は常にデータ本体の機械計測で裏取りする。** 累計値も「前回累計＋今回人数」の単純加算は重複分を二重計上する（実際に誤った）ため、毎回実測する。

## 5. コンテキスト・トークン効率 — メイン会話は司令塔に徹する

全34セッションの実測（Langfuse集計）で、トークン消費の96.6%は cache_read（毎ターンの文脈再送）であり、モデルの生成量は1%未満だった。つまり**コストはほぼ「メイン会話にどれだけ文脈を積んだか×残りターン数」で決まる**。ここから導かれる設計：

- **メイン会話で大きなファイルを Read しない。** データ本体（2.9MB）は jq/python 経由で必要部分のみ抽出。**Workflow の中間結果ファイルも同じ扱い**（本体には適用していたのに中間結果で適用し忘れ、900行超をメイン会話に載せた失敗がある）。
- **セッション序盤の大出力は複利で効く。** あるターンで載せた文脈はその後の全ターンで再送され続けるため、冒頭20〜30ターンでの大きな Read/Bash 出力が最も高くつく。検証スクリプトの出力は「一致件数＋不一致行のみ」を基本形にする（全一致行を print して24KBのツール出力になった失敗がある）。
- **エージェントへの入力は「メイン会話でしか用意できない情報」に絞る。** 既存レコードを args に埋め込むと構築だけで32KBをメイン会話が負担した。代わりに「各エージェント自身が jq で自分のレコードを取得する」方式（args には史料マッピングだけ渡す）にする。その際 jq は**必要フィールドだけの射影**にするとエージェント側の読み込みも減る。
- **本体調査は Workflow 1回に集約し、メイン会話は検証・マージのみ行う。** サブエージェントの cache_read はメイン会話に合算されないため、ターン数の複利を回避できる。最も軽かったセッションはこの構成（メイン会話の Read は3件のみ）だった。関連ファイルの読み込みも `cat a; echo ---; cat b` で1コマンド化するとターン数が減る。
- **失敗エージェントの再実行は「対象を絞った小さい新規Workflow」が安全。** resumeFromRunId は「同じ script+args ならキャッシュヒット」のはずだが、実際には16人×3回のフルリランが起きた実例（推定9.5Mトークンの無駄）や、配列先頭の変更で全再実行された実例がある。resume 後はエージェント実行数が想定と一致するか確認する。1〜2名の修正なら単発の小Workflowで済ませる。
- **「複数人物をまとめて1エージェントに長時間任せる」は人物分割より高くつく。** 出典マッピング調査を単発エージェントに任せたところ、1体で8.5Mトークン（69ターン）を消費し全237体中最大だった。15人を超えるようなまとめタスクは人物単位の並列Workflowに分割する。

## 6. 記録の運用 — 状態を1か所に、知見は都度その場で

- **進捗はチェックリスト（ドキュメント）とデータ本体のメタ情報を同じタイミングで更新する。** 片方だけ更新すると次のセッションが古い前提で動く。ブロック完了ごとに「データ反映・メタ追記・チェックボックス・特記事項の記録」をワンセットで行う。
- **実務知見はリポジトリのドキュメントに残す。** セッション用メモリはマシン間で共有されず、他の作業者（人間・AI とも）から見えない。史料マッピング・罠のチェックリストのような再利用情報は必ずリポジトリ側（本 docs/）に書き、メモリはそのポインタと経緯の記録にする。
- **判断根拠をデータに残す。** 境界事例の判定は結論だけでなく理由を各レコードの note に書く。後から基準を再検討するとき、監査するとき、サイトで「なぜこの数え方か」を説明するときの全てに効く。

## 7. 結果（この方法で達成できたこと）

- 364人×12項目（在位・死因・即位経路・改元・大赦・立后・皇太子廃立・遷都・親征・反乱鎮圧・被反乱・年齢）を4日間で悉皆確定。
- 終盤ブロックでは機械検証エラー（id不一致・count不一致・スキーマ外キー・論理逆転）が連続して完全0件。初期ブロック（id不一致20件超/35名）からの改善は、プロンプト設計と機械検証の積み上げによる。
- 判定品質は最後まで人物ごとの個別調査を維持（自動生成ゼロ）。極端値（隋煬帝の被反乱66件等）も原文引用に基づく個別計上で確定。
- 「調査済みだが不明」を正式な結果として区別したことで、完了検証（全員が全フィールドを保有）が機械的に可能になった。
