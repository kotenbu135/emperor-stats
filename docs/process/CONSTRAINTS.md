# 絶対に守るべき制約

## スクリプトによる自動生成禁止

**スクリプトで JSON データを自動生成することは禁止**です。過去2回失敗しています。

人物ごとに個別調査・判定した結果を書き込んでください。

### 計算補助スクリプトはOK

日数計算（ユリウス通日の差分など）のような **purely mechanical な計算補助** にスクリプトを使うのは可です。

**ただし**: 実際に調査・確定した日付を入力する場合に限ります。

### 禁止される使い方

- 史料からの人物リスト抽出をスクリプトに代行させる
- 判定そのものをスクリプトに代行させる

## データセットの一度きりの構築

`data/emperors.json` は一度きりの出力です。

- パイプライン化は行いません
- 自動再生成の仕組みは作りません

## 並行セッションとの共存（read-modify-write の徹底）

同じ作業ツリーで複数の Claude Code セッションが `data/emperors.json` を同時に編集することがあります（2026-07-21 のフェーズB調査で実際に複数回発生）。

- マージスクリプトは必ず「**実行直前に最新のファイルを読み込む**」「**対象 id のフィールドだけ更新する**」「**それ以外のレコード・キーには一切触らない**」を徹底する。ファイル全体を読み込んでメモリ上で書き換えて丸ごと書き戻す方式は、他セッションの並行成果を上書き消失させるリスクがある
- 作業完了時に `git status` / `git diff` で、自分が触っていないはずの id や `meta` ブロックが差分に含まれていたら、他セッションの並行作業の可能性が高い。自分の変更として説明したり、コミットに含めたりしない
- `task.md`・`docs/PROJECT_STATUS.md` のような進捗ドキュメントも同様に混在しうるため、自分が作業した範囲だけを追記する

## 外部 API への一括リクエスト禁止（小規模検証 → 明示的許可）

外部 API（Wikimedia 等）へ大量（100件超など）のリクエストを投げる作業は、いきなり全件実行しない。過去に肖像画調査で Wikimedia API へ364人分を並列実行して 429 と乱用警告を受けた事故があります。

- まず API のレート制限ポリシー（User-Agent 要件・推奨間隔）を確認し、可能なら**公式の一括データダンプ**（オフライン処理）を優先する
- ライブ API を使う場合は数件〜十数件でパイロットし、結果と想定リクエスト数をユーザーに提示して**明示的な許可を得てから**本実行する
- 429 やレート制限警告を受けたら、リトライ・バックオフで粘らず**即座に停止**して報告する
- 成功実績: Wikidata QID 紐付け（2026-07-21）は直列・maxlag=5・専用 User-Agent・計約12リクエストで 429 ゼロ

## データ正確性の最優先

**データの正確性が最優先です**。絶対に誤りのないデータが完成するまでサイト構築には着手しません。

`meta.status.overall` が `"in-progress"` の間は、公開・可視化には進みません。2026-07-18に364人全員・全12項目のデータ収集・検証が完了し `"completed"` になりましたが、この方針自体は今後データに誤りが見つかり訂正作業が必要になった場合にも引き続き適用されます。
