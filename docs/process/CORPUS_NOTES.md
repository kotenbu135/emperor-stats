# ローカルコーパス利用メモ

`china-history/`・`daizhigev20/`（[RESEARCH_PROCESS.md](RESEARCH_PROCESS.md)参照）を実際に調査で使う際の実務的な注意点・インデックスをまとめます。Claude の作業メモリにも同内容がありますが、マシン間で共有されないため、他の作業者や別環境で作業を再開する場合に備えてここにも記録します。

## コーパス検索のメモリ事故対策（`.{0,N}KW.{0,N}` 禁止・2026-07-24）

**史書調査中に WSL が落ちる（OOM）事故の主因は、コーパスに対する `grep -oE '.{0,40}(追尊|七廟|…).{0,40}'` 型のコンテキスト抽出パターンである。** 実測（2026-07-24）で確定した事実：

- **研究シェルの素の `grep` は GNU grep ではなく ugrep**（Claude Code が注入するシェル関数が同梱 ugrep を `-G --ignore-files --hidden` で起動するラッパー。`declare -f grep` で確認できる）。`/usr/bin/grep` が GNU grep、`rg` が ripgrep。
- **ugrep はこの `.{0,N}(巨大な漢字選択肢).{0,N}` 形を処理する時だけ爆発する**。再帰の有無は無関係で、**単一10MBファイル（明史.txt）でもピーク 4GB 超で強制終了（約37秒）**。トリガーは有界反復 `.{0,N}` で大きな Unicode 選択肢を挟む正規表現形状（ugrep の自動機構築が肥大）。
- 同じパターン・同じファイルで **`/usr/bin/grep`（GNU）＝3MB、`rg`＝20MB・0.02秒**。ヒット集合は3エンジンともバイト単位で一致（ugrep ≡ GNU grep は diff 0行）＝**品質劣化なし**。

**恒久ルール**（メイン会話・Workflow サブエージェント共通）：

1. **`.{0,N}KW.{0,N}` 形のコンテキスト抽出を素の `grep`／Grep ツールでコーパスに掛けない。** これが唯一の事故トリガー。前後文脈が欲しい時は `-n`（行番号）で位置特定 → `Read` の `offset`/`limit`、または `-C`（前後行）を使う。素のキーワード・選択肢検索（`.{0,N}` なし）は ugrep でも安全に完走する。
2. どうしても `.{0,N}` インライン抽出が必要なら **`/usr/bin/grep`（GNU・フラグ完全互換のドロップイン）か `rg` を明示指定**する。`rg` は最速だがフラグ体系が別（`-E` は「エンコーディング指定」で拡張正規表現ではない・POSIX でなく Rust regex）なので、grep フラグの無変換コピペは不可。
3. **範囲を絞る** — 単一書名ファイルか `_corpus_cache/*.txt`（数KB）を対象にし、コーパスツリーの再帰全走査をしない（本ドキュメントの行番号インデックス・キャッシュ機構がこのためにある）。
4. 重い検索を Bash で回す時は `( ulimit -v 4000000; … )` でハード上限を掛ければ、暴走してもエラー死して WSL 全体を巻き込まない（`ulimit` は Bash 子プロセスのみ有効・内蔵 Grep ツールの ugrep には効かない）。`.wslconfig` の `swap=16GB` は最後の安全網（前回 OOM 時の `swap=0` から改善済み）。

関連: [[wsl-crash-oom-claude-2026-07-23]]（メモリ側の記録）。

## china-history 側のファイル名・ラベルの罠

`china-history/` は正史ごとに本紀・列伝が章単位HTMLに分割されているが、ファイル名の巻数表記やラベルに書ごとの罠がある。新しい書に着手する前に確認する。

### 1. 列伝ディレクトリのファイル名巻数が絶対巻数ではなく相対番号になっている書がある

- **宋史**: 本紀47+志162+表32=241巻を差し引いた相対番号（例: 絶対巻243→列伝ファイル名は2）
- **金史**: 同様に本紀・志・表の巻数を差し引いた相対番号
- **明史**: 本紀24+志75+表13=112巻を差し引いた相対番号（例: 絶対巻122「韓林児伝」→ファイル名10、絶対巻123→ファイル名11）
- **元史**: 本紀ディレクトリは絶対巻数と一致（元史は本紀のみ47巻で完結するため例外）。列伝側は要確認。

対応: 新しい書に着手する際は、着手前に1〜2件を原文の巻数記載（「卷○○」等の見出し文言）と照合し、絶対番号か相対番号かを確定してからWorkflowプロンプトに明記する。

### 2. 「原文」ラベルのファイルが実際には白話訳（現代語訳）になっているデータ不備

実例: `china-history/晋书/帝纪/第十章--原文.html`（東晋・安帝紀＋恭帝紀相当）はラベル「原文」だが中身が白話訳。この巻は`daizhigev20/史藏/正史/晋书.txt`を代わりに使用した。

対応: 新しい巻を原文として引用する前に、文言文（漢文調、断句や虚字の使い方）であることを目視確認する。文体が現代語調（「的」「了」「是」の多用等）なら白話訳の取り違えを疑い、daizhigev20側で裏取りする。

### 3. china-historyに収録がなく daizhigev20 が必須になる書

清史稿・資治通鑑・西夏書事・十国春秋等（china-historyは二十四史＋新旧五代史のみ収録、これらの準正史・編年体史料は非収録）。

対応: 着手前に対象の書が二十四史＋新旧五代史に含まれるか確認し、含まれなければ最初からdaizhigev20（行番号インデックス化が必要な場合は下記参照）を使う前提で計画する。

## 皇帝ごとの原文キャッシュ（`_corpus_cache/`）

2026-07-17、トークン効率化のため、同一人物の原文を複数の調査グループ（改元・親征・遷都・年齢等）で何度も読み直す無駄を減らす目的で、`scripts/build_corpus_cache.py`を作成した。china-history（HTML）・daizhigev20（txt）から人物ごとの本紀原文を抽出し、HTMLタグ・ナビゲーション文言を除去した上で`_corpus_cache/<id>.txt`に書き出す。判定・調査を伴わない機械的なテキスト抽出のみで、出典・巻の同定自体は本ドキュメントのインデックスに基づく。

`_corpus_cache/`は`.gitignore`対象（china-history/daizhigev20と同様、ローカルコーパスから決定論的に再生成可能なため）。再生成する場合は`python3 scripts/build_corpus_cache.py`を実行する。新しいブロックに対応する際は、このスクリプトに書名・巻・行範囲マッピングを追記していく。

**現在対応済み**: 明（16名）・清（11名、宣統帝は清朝期1908-1912のみ）・前漢（14名）・後漢（14名）・五代十国（34名）・宋遼西夏金（52名）・元/北元/元末群雄（18名）・秦/新末群雄/玄漢/西晋/五胡十六国/南北朝（48名、グループ5単独ブロック）・三国/両晋/十六国追加/南朝/北朝/隋/隋末群雄/唐（149名、task.md 3-1 フェーズB B-2、2026-07-21）。364名（`zhonghuadiguo-yuanshikai`＝袁世凱のみ正史範囲外のため対象外）全員分のキャッシュが揃っている。範囲は上記の明史・清史稿の表、下記の前漢・後漢の表、末尾の五代十国の表・宋遼西夏金の表・元/北元/元末群雄の表・グループ5単独ブロックの節、および本節末尾の「B-2ブロック」節に対応。
