# データ調査の進め方

## 大原則

**判定の根拠は必ず一次史料（正史の本紀・列伝）の原文に置きます。** 二次的な解説サイト・検索結果の要約サマリーだけを根拠に判定してはいけません。

WebSearch の返り値は AI による要約であり、原典の直接引用か二次創作記事の要約かを区別せずに使うと誤りが混入します。

### 実例

後漢・沖帝の死因調査で、次代の質帝の記述との混同を含む要約をそのまま採用してしまい、後に原典検証で誤りと判明しました。遡及検証時（2026-07-15）に秦・前漢・玄漢・後漢 32 人分を Workflow で並列に再検証し、5 件の誤り・精度低下を修正しました。

## 調査手順

### 1. 候補人物の洗い出し（完了・2026-07-16）

在位データ（`meta.status.phases.reignData`）調査は秦〜清の全ブロックで調査し候補人物の洗い出します。

### 2. 個別事実の確認

生前称号が本当に皇帝だったか、即位日・崩御日、死因など、各人物について個別の事実を一次史料の原文で確認します。

以下のローカルコーパスを優先し、それでも確定できない場合のみ維基文庫を参照します。

#### 第一選択: ローカルコーパス（`china-history/`・`daizhigev20/`）

事前に取得済みの正史全文ローカルコピーをまず参照します。ネットワークアクセスが発生しないため、繰り返し参照しても外部サーバーに負荷をかけません。

- **`china-history/`**（[hunterhug/china-history](https://github.com/hunterhug/china-history) を `git clone --depth 1` で取得済み）: 正史ごとのディレクトリ配下に、本紀・列伝の章単位で「原文/段訳/译文」のHTMLファイルがある。例: `china-history/史记/十二本纪/第一章-五帝本纪-原文.html`
- **`daizhigev20/`**（[garychowcmu/daizhigev20](https://github.com/garychowcmu/daizhigev20) を `git clone --depth 1` で取得済み）: `史藏/正史/` 配下に書ごと1ファイルのプレーンテキスト。例: `daizhigev20/史藏/正史/史记.txt`。章単位に分割されていないため該当篇を検索（grep等）して該当箇所を探す。

両コーパスとも詳細は各ディレクトリの `CACHE_INFO.md` を参照してください。ファイル名の巻数の罠・行番号インデックス等の実務的な注意点は [CORPUS_NOTES.md](CORPUS_NOTES.md) にまとめています。

**重要**: これらはOCR/転記由来の誤字を含む可能性があるコピーです。ここから直接引用する場合も、可能な範囲で文脈（前後の記述）と照合し、不自然な文字は誤字を疑ってください。

##### `daizhigev20/` 大容量ファイルの行番号インデックス化（コンテキスト効率）

`daizhigev20/` は書ごと1ファイルで章分割がなく、`清史稿.txt`（15.6MB）・`资治通鉴.txt`（9.6MB）等は書全体をエージェントに読ませると無駄なトークン消費になります（china-history側に収録がなくdaizhigev20が必須になる書: 清史稿・資治通鑑・西夏書事・十国春秋等）。

これらの書は「　　本纪一　太祖本纪」のような本紀見出しが規則的に入っているため、Workflow起動前に1回だけ `grep -n "本纪\|帝纪\|载记" ファイル名` 等で見出し行を洗い出し、皇帝ごとの開始行番号インデックスを作ります。各調査エージェントには`sourceFiles`に加えてその人物の行範囲（次の見出しの直前まで）を渡し、書全体ではなく `Read` の `offset`/`limit` でその範囲だけを読ませます。原文は省略・要約せず全文そのまま読めるため判定精度は変わりません。

このインデックス化は「史料からの人物リスト抽出」や「判定そのもの」をスクリプトに代行させるものではなく、原文の中で何行目から何行目を読むかという機械的な範囲指定にすぎないため、[CONSTRAINTS.md](CONSTRAINTS.md)の禁止事項には抵触しません。

#### 第二選択: ローカルコーパスで確定できない場合 → 維基文庫（`zh.wikisource.org`）

ローカルコーパスに該当篇がない、篇が欠落している、あるいは文字化け等で判定不能な場合のみ、WebFetch で維基文庫（中文維基文庫）に直接アクセスします。

**サーバー負荷への配慮（必須）**:
- 同じページへの重複アクセスを避ける（一度取得した本文は会話内で使い回す）
- 短時間に多数の異なるページへ連続アクセスしない。Workflow を使う場合も並列度・リクエスト間隔に注意する
- 必要な篇だけに絞ってアクセスし、探索目的での総当たりアクセスは行わない

#### 第三選択: 原文に直接アクセスできない場合

WebSearch で「[人物名] 死因/即位 史記/漢書/後漢書等 原文」のように検索します。

**重要**: 検索結果の中から**原文の直接引用（漢文）を含むもの**を選んで採用します。
- 個人ブログ・まとめサイトの解説文だけを根拠にしません
- そこに原文が引用されていればそれを確認します

日本語版・中国語版 Wikipedia（`baike.baidu.com` 含む）は候補の見取り図・第一次検索の入口として使ってよいですが、原文への到達を必ず試みます。

#### ページが404 の場合

曖昧さ回避や別表記を試します。例: `李寿_(成漢)`

### 3. 日付の特定

即位日・崩御/退位/廃位/処刑日を可能な限り日付レベルで特定します。

**複数の情報源で日付が食い違う場合**:
- 原典（`梁書`、`資治通鑑` など）に近い情報や infobox の記載を優先
- 確認できない精度をでっち上げません
- `datePrecision` で `year`/`month`/`day` を正直に記録します

### 4. 判定結果の提示

判定結果（含める/除外とその根拠、死因等の追加スキーマ項目の判定根拠）をブロック単位でユーザーに提示し、承認を得てから `data/emperors.json` に追記します。

**原文の引用**: 判定に使った原文の直接引用は `note`/`source.note` 等に残します。要約だけでなく原文が確認できる形にします。

### 5. フェーズごとの完了記録

`meta.completedBlocks`（在位データ）・`meta.deathCauseCompletedBlocks`（死因）等、フェーズごとの完了記録に以下を記載します：
- 完了したブロック名
- 除外理由
- 保留事項
- 原典で修正した誤り

### 6. 遡及検証

既に入力済みのデータであっても、WebSearch の要約のみを根拠にしていたことが判明した場合は、原典で遡及検証します。

## コンテキスト効率（メイン会話で `data/emperors.json` 全体を読まない）

`data/emperors.json` は2.9MB・6万行超あり、メイン会話（Workflow起動前後のやり取り）で Read すると毎回大量のトークンを消費します。**メイン会話では原則としてこのファイルを Read しません。** 以下のように Bash 経由で完結させます。

### 対象人物の抽出（Workflow の `args` を作る）

`jq` または `python3 -c` のワンライナーで該当ブロック（`dynasty.section`等の条件）に絞り、Workflow に渡す必要最小限のフィールド（`id`/`name`/`dynasty`/`reigns`の要点/`source.page`等）だけを抽出し、標準出力をそのまま `args` に渡します。ファイル全体をメイン会話のコンテキストに乗せません。

### Workflow 結果のマージ

Workflow の結果（JSON）を一時ファイルに保存し、Python スクリプトで `data/emperors.json` を読み込み→該当 `id` のオブジェクトだけ更新→書き戻します（[CONSTRAINTS.md](CONSTRAINTS.md) の「計算補助スクリプトはOK」の範囲内）。メイン会話ではスクリプトの実行結果（成功件数・エラー有無）だけを確認し、ファイル内容そのものは読み書きしません。

#### マージスクリプトに組み込む機械的バリデーション（毎回目視チェックしない）

これまでのブロック（南朝・北朝・隋唐・五代十国・宋遼西夏金・元など）では、マージ後に人力で「`count == events.length` の一致」「`note_extra: null` 等の余計なキーの紛れ込み」を毎回目視確認していました。判定内容（史実の正誤）には関与しない純粋な構造チェックなので、マージスクリプト自身に以下を組み込み、違反時はスクリプトがエラー終了する（マージさせない）運用にします。これにより目視チェックとメイン会話でのやり取りが不要になります。

- **キーのホワイトリスト化**: 対象フィールド（`eraChangeCount`/`amnestyCount`/`empressInstallationCount`/`crownPrinceDepositionCount`/`capitalRelocationCount` は `count`/`events`/`confidence`/`note`、`personalCampaignCount`/`rebellionSuppressionCount`/`rebellionSufferedCount` は `ADDITIONAL_SCHEMA.md` 記載の該当キーのみ）に含まれないキーは自動的に除去してからマージする。
- **`count == len(events)` の突き合わせ**: 不一致ならその人物 `id` を含めてエラー終了し、マージを中断する（`count` を判明件数の最小値として意図的に不一致にするケースは `note` に不確実性を明記した上でスクリプト側に許容リストとして明示的に渡す）。
- **`events[]` 内の日付precisionの整合**（`date`/`datePrecision` の片方だけが `null` になっていないか等）も同様に機械チェック対象に含めてよい。

このバリデーションは「判定そのものをスクリプトに代行させる」ものではなく、既に人力で確定した調査結果の構造的な後処理チェックに過ぎないため、[CONSTRAINTS.md](CONSTRAINTS.md) の禁止事項には抵触しません。

### ユーザーへの提示

ブロック完了報告は追加分の要約（人物名・値・特筆事項）で行い、`data/emperors.json` 全体や大きな範囲の cat/Read は避けます。差分確認が必要な場合は `git diff -- data/emperors.json` を使います。

## Workflow エージェントの id を固定する（結果マージの手戻り防止）

複数人物を並列調査する Workflow では、エージェントが返す JSON の `id` フィールドが、こちらが指定した ID と異なる人名ベースのスラッグ（例: `zhang-zuo` → 正しくは `qianliang-zhangzuo`）を自己申告するケースが頻発していました（南朝35名中20件超、北朝34名中8件など）。これに気づかず位置ベース（journal.jsonl の `started` イベント順など）で結果を対応付けると誤マッチが発生します（実例あり）。

これを防ぐため、Workflow の各エージェントへのプロンプトには次を必ず含めます：

- 「返す JSON の `id` フィールドは、このプロンプトで指定した ID `<id>` を一字一句そのまま使うこと。人物名からの独自生成・変更は禁止」
- 対象人物の `id` を `args` の該当エントリに明記し、プロンプトのテンプレート内で `${item.id}` のように直接埋め込む（人物名から `id` を推測させない）

これでも不一致が発生した場合のみ、`agesInfoFound.deathDate` 等の内容照合による手動 remap にフォールバックします（判定精度に関わる作業ではないため、メイン会話ではなく一時ファイル＋Python スクリプトで機械的に照合・remap して構いません）。
