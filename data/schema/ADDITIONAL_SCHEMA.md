## 情報源・調査方針（共通）

[DEATH_CAUSE_SCHEMA.md](DEATH_CAUSE_SCHEMA.md) と同じ方針に従う。原典（正史の本紀・列伝）を第一次情報源とし、WebSearchの要約のみを根拠に確定しない。[docs/process/RESEARCH_PROCESS.md](../../docs/process/RESEARCH_PROCESS.md) を参照。

- グループ2（改元・大赦・立后・皇太子廃立）は本紀の通読で4項目同時に埋める。
- グループ3（親征・反乱鎮圧・被反乱）は本紀の軍事記事を数える。**このドキュメントで定めた数え方の基準に従う**（下記参照）。

## 回数系指標の計上期間（本人の実権掌握期・2026-07-22 明文化）

改元・大赦・立后・皇太子廃立・遷都・親征・反乱鎮圧・被反乱の8つの回数系指標は、**その人物自身が君主として実権を掌握していた期間（実権掌握期）に本人が行った行為**を計上する。皇帝号を正式に称する前でも、王・天王・僭号などの称号のもとで君主権を行使していた期間の行為を含める。

- **理由**: 石虎（後趙、長く「趙天王」）・李元昊（西夏、称帝前は「西平王」）・劉淵（漢、「漢王」期）・楊溥（呉、「呉王」期）・黄巣（「王」自称期）など、建国者は皇帝号を称する前から君主として改元・大赦等を行っており、皇帝在位期間のみに限ると統治実態を著しく過小評価する。各フィールド定義中の「（皇帝）在位中」という表現は皇帝号確立後に限る趣旨ではなく、**本人が君主権を行使した期間**を指す（この運用を 2026-07-22 に確定・明文化。8フィールドすべてに一律に適用し、フィールドによって計上期間を変えることはしない）。
- **他者の行為は含めない**: 本人でなく先代・父などが行った行為は、後にその人物が皇帝となっても計上しない（例: 前燕・慕容儁の遷都から、父・慕容皝が燕王期に行った 0341 年の龍城遷都を除外した。2026-07-22。原文『晋書』載記第九「皝遷都龍城」）。
- 該当する称帝前イベントには `note` に【皇帝即位前】等の明示を付す。CI（`scripts/validate_emperors.py` の `check_event_reign_range`）は在位 ISO 年範囲外に出るこれらを `KNOWN_PREACCESSION_EVENTS` として許容する（在位範囲内に落ちる称帝前イベントは検出対象外だが同じ方針で計上する）。
- 経緯: 2026-07-22 のデータ QA で、称帝前イベント計上を「遷都のみ在位中に限定・他は包括」とする折衷案が一度検討されたが、改元・遷都とも定義文言が同一（ともに「在位中」）で折衷に原理的根拠がないため撤回し、全フィールド一律「本人の実権掌握期」で統一した。

## 1. 即位経路（`accessionRoute`）

個人ページ・本紀の即位記事を読めば判明する。死因調査（グループ1）とセットで調査すると効率的。

```json
"accessionRoute": {
  "category": "世襲",
  "note": "自然文での説明（先帝との関係、擁立者の有無など）",
  "confidence": "high",
  "source": { "page": "...", "lang": "ja" }
}
```

- `category`: 下記カテゴリ一覧から単一値（enum）。
- `note`: 経緯を自然文で記述（例：「先帝の嫡長子として遺詔により即位」「宦官○○に擁立され即位」）。
- `confidence` / `source`: `deathCause` と同じ運用。

### カテゴリ一覧

| 値 | 定義 |
|---|---|
| `世襲` | 先帝の子・兄弟など血縁者が、皇位継承の通常の手続き（遺詔・冊立太子からの継承等）で即位 |
| `簒奪` | 臣下・軍閥・外戚などが実力・クーデターにより先帝（または先王朝）から皇位を奪って即位 |
| `禅譲` | **前王朝（別姓・別政権）の皇帝**から形式上・儀礼上の「譲位」を受けて即位＝王朝交代を伴う受禅（実態が簒奪に近い禅譲劇も含み、その場合は `note` に経緯を明記）。例：曹丕（後漢→魏）・司馬炎（魏→西晋）・楊堅（北周→隋） |
| `内禅` | **同一王朝内**で、先帝が在世のまま行う生前譲位により即位（父子間が典型だが、兄弟・傍系宗室への譲位も含む）。強要された譲位や権臣・宦官主導の内禅も、譲位の形式を踏んでいれば含める（その場合は実態を `note` に明記）。例：唐玄宗（睿宗の生前譲位）・南宋孝宗（高宗の内禅）・清仁宗（乾隆の内禅）・金末帝（哀宗の伝位） |
| `擁立` | 臣下・軍閥・宦官・外戚等の主導により、本人の主体的な簒奪行為なしに即位させられた（傀儡的即位を含む） |
| `復位` | 一度廃位・退位した後、再び即位した（`reigns[].isRestoration: true` の在位に対応。初回即位の経路は別途 `category` で判定し、`note` に復位の経緯を記す） |
| `建国` | 王朝・政権を新規に樹立して自ら皇帝を称した（先行政権からの継承がない場合。五胡十六国や反乱政権の初代に多い） |
| `不詳` | 即位の経緯が原典に見当たらない |
| `諸説あり` | 複数の原典/通説が対立し一つに絞れない |

### 設計判断

- **`selfProclaimed: true` の人物でも `世襲`/`復位` になり得る**（例：ある政権内での2代目以降は父からの世襲）。`accessionRoute` は「その人物個人の即位経路」であり、`flags.selfProclaimed`（政権全体の正統性）とは独立した軸。
- **禅譲と簒奪の境界**：形式上の譲位儀礼を経ていれば `禅譲`、実力行使のみで即位すれば `簒奪`。両者が併存するケース（禅譲の形式を取った事実上の簒奪、例：曹丕・司馬炎）は `禅譲` を採用し、`note` に「実質的には簒奪」である旨を明記する（史書の記述上の建前と実態を区別して残す）。
- **`禅譲` と `内禅` の境界（王朝交代の有無）**：譲位により即位した場合、譲位元が**別姓・別政権（前王朝）**なら `禅譲`（＝王朝交代を伴う受禅）、**同一王朝内**の先帝からの生前譲位なら `内禅`。この2つは史学的に別概念（禅譲＝堯舜・曹丕型の易姓、内禅＝父子等への生前譲位）であり、旧スキーマは両者を `禅譲` に混在させていたため2026-07-22に `内禅` を分離した。
- **`内禅` と `世襲` の境界（先帝の生死）**：先帝が**在世のまま譲位した**（太上皇となる等）＝ `内禅`、先帝の**崩御に伴う継承**（遺詔・立太子からの践祚）＝ `世襲`。受け手が正規の皇太子・嫡子であっても、生前譲位で即位したなら `内禅` とする（例：北魏孝文帝・清仁宗は皇太子だが献文帝・乾隆の生前譲位のため `内禅`）。養子への生前譲位（南宋孝宗＝高宗の養子）も `内禅`。
- **`内禅` と `擁立` の境界（譲位行為の有無）**：先帝自身が譲位の行為を行った（強要・宦官主導でも、詔や璽の授受という譲位形式を踏む。例：唐憲宗＝順宗の内禅を宦官が主導）なら `内禅`、先帝が譲位行為を行わず第三者が新帝を立てた（例：南宋寧宗＝光宗は病で譲位行為をせず大臣が擁立、正史は「内禅」と粉飾するが実態は擁立）なら `擁立`。
- **`復位` が優先**：一度廃位・退位した後の再即位は、たとえ先帝の生前譲位（内禅）を伴っても `復位` を採用し、内禅の経緯は `note` に記す（例：武則天の生前譲位を経た唐中宗・睿宗の第2次在位は `復位`）。
- **複数回即位した人物**：`reigns` 配列に対応させず、`accessionRoute` はレコード直下に単一値で持たせる（初回即位の経路を主とし、復位がある場合は `category: "復位"` を採用、初回の経路は `note` に記す）。理由：即位経路は「その人物の皇帝としてのキャリアの始まり方」を表す代表値としての性格が強く、`deathCause` 同様レコード単位の集約情報として扱うほうが `reignSummary` の設計思想と一貫する。

## 2〜5. 本紀通読カウント系（改元・大赦・立后・皇太子廃立）

いずれも「回数（`count`）＋明細配列（`events`）」の同じ構造を持つ。本紀を1回通読すれば4項目同時に埋められる。

```json
"eraChangeCount": {
  "count": 3,
  "events": [
    { "date": "-0133-01-01", "datePrecision": "year", "note": "建元→元光に改元（元光元年＝前134年）", "source": { "page": "...", "lang": "ja" } }
  ],
  "confidence": "high",
  "note": "総括コメント（不確実な件数がある場合など）"
}
```

- `count`: `events` 配列の要素数と一致させる（一致しない場合は不確実な件数が別途あることを意味し、`note` で説明する）。
- `events[].date` / `.datePrecision`: `reigns[].startDate`/`datePrecision` と同形式。日付が特定できない場合は `date: null`, `datePrecision` は省略可。
  - **紀元前の年規約（2026-07-22 統一）**: `reigns` と同じ ISO 8601 天文年（前n年 → `-(n-1)`。例: 前134年 → `-0133`）で表記する。歴史年直記（前n年 → `-n`）は使わない。month/day precision では実際のユリウス暦時点が属する年を使う（漢初〈太初改暦・前104年より前〉は年始が冬十月のため、冬十月〜十二月・閏九月のイベントは紀年の対応年より1年前の ISO 年に落ちる。例: 文帝元年冬十月 → `-0179-10`）。year precision では紀年（元号の元年等）の対応年を使う。`scripts/validate_emperors.py` の `check_bce_event_years()` が在位 ISO 年範囲チェックと note の「前n年」明記との突合を行う。
- `events[].note`: 自然文で事由を記す。
- `confidence`: レコード全体（`count` の確からしさ）に対して `high`/`medium`/`low`。
- 該当なし（0回）の場合も `count: 0`, `events: []` で明示する（フィールド自体を省略しない。調査未実施と0回を区別するため、未調査の人物は `phases` の `status` で管理し、フィールド自体を付与しない）。

### 2. 改元回数（`eraChangeCount`）

- 「1回」＝新しい年号（元号）への切り替え1回。即位に伴う最初の建元は含めるか要判断が分かれるため、**即位に伴う最初の元号制定も1回としてカウントする**（皇帝在位中に何種類の年号を使ったか、という実質を反映するため）。`events[0]` に即位時の建元を含める。
- 年号を持たない王朝・時代（改元制度が未整備な秦など）は `count: 0` とし `note` に「年号制度なし」等と明記する。

### 3. 大赦回数（`amnestyCount`）

- 「1回」＝本紀に大赦（「大赦天下」等）の記事として明記された1件。部分的な恩赦（特定地域・特定罪状のみの減刑）は含めず、全国規模の「大赦」と明記されたものに限定する。
- 判定基準の詳細な線引きが必要な曖昧ケースが出た場合は `verification.notes` 相当として `note` に残し、実調査開始時にユーザーに確認する。

### 4. 立后（皇后冊立）回数（`empressInstallationCount`）

- 「1回」＝皇后として正式に冊立された1件。同一人物の再冊立（廃后後の復位）も別カウントとする。皇后不在のまま在位した場合は `count: 0`。
- `events[].note` に冊立された皇后の名（分かれば）を記す。

### 5. 皇太子廃立回数（`crownPrinceDepositionCount`）

- 「1回」＝立てられていた皇太子（または皇太弟等の法定推定継承者）を廃した1件。立太子そのものの回数ではなく**廃立（廃止）のみ**をカウントする（フィールド名どおり）。
- `events[].note` に廃された皇太子の名・理由を記す。

## 6. 遷都回数（`capitalRelocationCount`）

個人レコード単位で「自分の在位中に何回遷都したか」を持たせる（王朝単位の集計は将来サイト側で `dynasty.name` によりグループ集計する）。

```json
"capitalRelocationCount": {
  "count": 1,
  "events": [
    { "date": "-0210-01-01", "datePrecision": "year", "from": "咸陽", "to": "洛陽", "note": "..." }
  ],
  "confidence": "high",
  "note": ""
}
```

- 構造はグループ2の4項目と共通（`count` + `events`）。`events[].from`/`.to` に遷都前後の都名を追加する点のみ異なる。
- 同一王朝内の先代皇帝が定めた都をそのまま使い続けた場合は `count: 0`。

### 「1回」の数え方の基準（2026-07-17確定）

- **正式な遷都（恒久的な主都機能移転）のみカウントする。** 戦乱による一時的な避難・行幸（例：唐玄宗の蜀行幸、北宋徽宗・欽宗の金への連行、南宋高宗初期の転々とした行在）は、恒久的な遷都と認められない限りカウントしない。
- **副都・陪都制度は対象外。** 唐の東都洛陽、元の上都、遼・金の五京制など複都制の王朝で副都を新設・行幸しても、正式な「主都」自体が変更されない限りカウントしない。ただし主都機能が事実上恒久的に移った場合（例：武則天による洛陽への遷都、南宋の臨安）は例外的に主都変更とみなしカウントする。
- **建国時の最初の定都は対象外。** 先行する自王朝の都からの移転ではなく、新王朝の初代皇帝が新規に都を選定した場合（`events[].from` に相当する自王朝の先代都が存在しない場合）は `count` に含めない。
- **短期間の連続移転はそれぞれ独立カウントする。** 同一皇帝の在位中に複数回都が変わった場合（例：後漢献帝期の洛陽→長安→許昌）は、一連の戦乱下の移動であっても `events` に個別に記録し、それぞれ1回として数える。

### 調査方法（トークン効率、2026-07-17確定）

グループ2・3の「人物ごとに1エージェント」方式は遷都には非効率（364人中大半が`count: 0`になる離散的な王朝単位の事象のため）。代わりに以下の2段階方式を採る。

1. **王朝（群）単位で遷都イベント年表を先に確定する**（グループ2/3と同じ12ブロック区分を流用し、複数政権が併存するブロック（南北朝・五胡十六国・五代十国等）はさらに王朝ごとに細分化してよい）。各エージェントは「その王朝で正式な遷都が何回あったか」を本紀から調査し、`date`/`datePrecision`/`from`/`to`/出典を伴うイベントリストのみを返す（人物への割当はこの時点で行わない）。
2. **既存の `reigns[].startDate`/`.endDate`（在位データ調査で確定済み）を使い、イベントの日付がどの皇帝の在位期間に該当するかをPythonスクリプトで機械的に割り当てる**。該当なしの皇帝には `count: 0, events: []` を一括付与する（`note`はテンプレ文言で可）。日付が皇帝交代の直前直後で判定が割れる境界事例のみ、個別に追加確認する。

この方式により、実際のLLM呼び出しは「王朝(群)単位のイベント列挙」十数〜数十件程度で完結し、364人分の個別調査より大幅に少ないトークン消費で済む。マージ時のホワイトリスト検証・`count == events.length`チェックは既存の運用（`merge-validation-automation`）をそのまま適用する。

## 7〜9. 軍事系カウント（親征・反乱鎮圧・被反乱）

判定基準を先に確定してから調査に着手する方針で進めた（経緯は [docs/PROJECT_STATUS.md](../../docs/PROJECT_STATUS.md) グループ3の節を参照）。基準は以下の通り確定：

### 7. 親征回数（`personalCampaignCount`）

- **「1回」＝同一の目的地・同一の相手に対する一連の軍事行動（出征〜帰還または現地での終結まで）を1回とする。** 同一遠征中に複数の敵を破っても1回、年をまたいでも同一目的の継続なら1回。年を空けて同じ相手に再度出征した場合は別カウント。
- 親征＝皇帝自身が軍を率いて戦場に赴いた場合のみ。将軍への派遣・勅命のみ（皇帝本人が出陣しない）は含めない。

```json
"personalCampaignCount": {
  "count": 2,
  "events": [
    { "startDate": "-0200-01-01", "endDate": "-0200-12-31", "datePrecision": "year", "target": "匈奴", "outcome": "敗北（白登山の戦い）", "note": "..." }
  ],
  "confidence": "high",
  "note": ""
}
```

- `events[].target`: 遠征相手（民族名・政権名・人物名など）。
- `events[].outcome`: 自然文または短句で結果を記す（勝敗が不明瞭な場合は「不詳」）。

### 8. 反乱鎮圧回数（`rebellionSuppressionCount`）

- **「1回」＝独立した首謀者・独立した蜂起地点を1件とする。** 同じ反乱が複数地域に飛び火しても首謀者が同一なら1件、鎮圧に複数年かかっても1件。
- 皇帝側（政権側）が鎮圧の主体だった件数。実際に鎮圧に成功したか否かは問わない（鎮圧を試みて失敗し王朝崩壊に至った場合もカウントし、`note` に結末を記す）。

```json
"rebellionSuppressionCount": {
```

### 9. 被反乱回数（`rebellionSufferedCount`）

- 数え方の単位は反乱鎮圧回数と同じ（首謀者/蜂起単位＝1件）。
- 皇帝に対して起こされた反乱の件数。鎮圧の成否は問わない。自身が反乱により廃位・殺害された場合もその反乱をカウントし、`deathCause`（`category: "暗殺"`/`"戦死"` 等）や退位経緯（`reigns[].note`）と内容が重複してよい（別軸の集計のため重複を許容する）。
- **`rebellionSuppressionCount` と `rebellionSufferedCount` は原則として同一の反乱群を両面から集計したものであり、通常はほぼ同じ件数・同じ反乱が両方に計上される（2026-07-16確定）。** 反乱者の社会的身分・政治的地位（無位無官の農民か、諸侯王か、丞相か等）や規模の大小は問わない。飢民・盗賊による蜂起（例: 黄巾の乱、赤眉の乱）であっても、実際に武装し政権への反抗行動（挙兵）を取り、政権側がそれを鎮圧の対象とした（＝`rebellionSuppressionCount`に計上した）のであれば、原則として`rebellionSufferedCount`にも同じ反乱を計上する。「反乱の標的が皇帝個人の廃位・弑逆を意図していたかどうか」を被反乱算入の判定基準にしてはならない（反乱の目的が地方での自立・自衛・略奪等であっても、皇帝の統治権に対する武力反抗である限り被反乱に含める）。
- 両者が食い違ってよい正当な理由は以下に限定する。これ以外の理由で一方のみに計上する場合は誤りを疑うこと：
  1. **集計単位の粒度差** — 鎮圧側は複数の首謀者・地域を1つの軍事行動として合算したが、被反乱側は首謀者ごとに個別カウントした（またはその逆）場合。
  2. **未鎮圧のまま在位終了** — 反乱が皇帝の在位終了（崩御・退位）までに鎮圧行動が着手・完了しなかった場合、`rebellionSuppressionCount`には含めなくても`rebellionSufferedCount`には含める。
  3. **皇帝自身が攻撃の主体だった場合** — 皇帝が権臣・宦官等を先制的に武力で打倒した場合（相手はまだ反乱を起こしていない）は、反乱そのものが成立していないため両方から除外する。
  4. **朝廷内部の権力者による兵力を伴う弑逆・クーデター** — 丞相・宦官・外戚等が兵力を用いて皇帝を弑逆・廃位に追い込んだ場合は`rebellionSufferedCount`に含めるが、鎮圧の主体が皇帝側ではなく反乱者側であるため`rebellionSuppressionCount`には計上しない（例: 望夷宮の変）。
- 対象が皇帝の統治下にない独立勢力・対等な群雄との抗争（統一戦争）や、服属していない外国勢力との対外戦争は、鎮圧・被反乱いずれの対象にもならない（`rebellionSuppressionCount`の基準と対称）。ただし辺境の郡県・属国等、皇帝の統治機構下にある集団（例: 属国の異民族）による蜂起は「服属民の反乱」として両方に含める。
- 実際の兵力行使（挙兵）に至らず、密告・摘発のみで終わった謀反計画は、鎮圧・被反乱いずれにも含めない（この基準は変更なし）。

```json
"rebellionSufferedCount": {
  "count": 1,
  "events": [
    { "startDate": "0755-01-01", "endDate": null, "datePrecision": "year", "leader": "安禄山", "name": "安史の乱", "outcome": "退位に追い込まれた", "note": "..." }
  ],
  "confidence": "high",
  "note": ""
}
```

- `rebellionSuppressionCount` と `rebellionSufferedCount` は独立フィールド。同一の反乱イベントが（鎮圧に関わった別の皇帝の）`rebellionSuppressionCount` と（対象にされた皇帝の）`rebellionSufferedCount` の両方に登場することは想定内であり、突き合わせの整合性チェックは行わない。
- **朝廷内部の権力者（丞相・宦官・権臣・近衛兵など）による、実力（兵力）を伴う弑逆・クーデターも `rebellionSufferedCount` に含める**（2026-07-16確定）。首謀者が外部の蜂起勢力か内部の権力者かは問わない。「兵力を用いて皇帝を弑逆・廃位に追い込んだか」を基準とする（例: 秦二世に対する望夷宮の変）。単なる暗殺未遂（刺客による単独犯行等、組織的な兵力を伴わないもの）は含めない。

## 10. 即位時年齢・没年齢（`ages`）

グループ1〜3（即位経路・死因・改元・大赦・立后・皇太子廃立・親征・反乱関連）調査の副産物として、2026-07-17時点で364人中213人分がすでに部分的に埋まっている（機会的補完の運用ルール、下記参照）。2026-07-17、能動的な悉皆調査の対象に正式に格上げした（`meta.status.phases.ages.status` は `deferred` から着手ステータスに変更する）。

**運用ルール（継続）**: グループ1〜4調査中に読んでいる原典・列伝の中で生年月日・没年月日・即位時年齢・没年齢が判明した場合は、その都度このフィールドを埋める（これに加えて、専用の調査ブロックでも悉皆的に埋めていく）。

```json
"ages": {
  "birthDate": "-0259-01-01",
  "birthDatePrecision": "year",
  "deathDate": "-0210-09-10",
  "deathDatePrecision": "day",
  "accessionAge": 13,
  "deathAge": 49,
  "confidence": "medium",
  "note": "生年は諸説あり。"
}
```

- `birthDate`/`deathDate`: ISO8601、不明部分は `null`。`deathDate` は原則 `reigns` の最終 `endDate` と一致するはずだが、退位後に別途没した場合はそちらを優先する（機械的な自動転記はせず、`deathCause` の記述で個別に確認する）。
- `accessionAge`/`deathAge`: **数え年（中国伝統の年齢計算）で統一する（2026-07-17確定）**。既存213人分の実データはすでに数え年で統一されており（原文の「時年◯◯」「年◯◯」等の直接記載を数え年としてそのまま採用）、この既存実績に合わせる。原文に満年齢由来の記載がある場合は `note` にその旨を明記した上で数え年に換算するか、換算不能なら `confidence: low` として両論併記する。
- **直接記載を最優先、次点で確度の高い逆算値を採用する**（三国ブロックで確立済みのルール）。同一人物について「特定の年号年における年齢」が原文に直接記載され、かつ知りたい時点（即位年・没年等）の年号年も確定している場合、両者の**年号年ラベルの差分**を直接記載の年齢に加減して逆算してよい（暦月をまたぐ誤差が生じないため）。逆算値は可能な限り別の独立した直接記載値とクロスチェックし、不一致の場合は無理に一方を採用せず `note` に不一致を明記して該当フィールドの追加自体を保留する。
- **生年・没年齢が判明しない人物も、調査済みであることを明示する。** 調査した結果、原典に手がかりが見当たらなかった場合は、`ages` オブジェクト自体は付与し、`birthDate`/`deathDate`/`accessionAge`/`deathAge` は `null` のまま、`note` に「調査したが原典に生年月日・享年の直接記載が見当たらなかった」旨を記す（「未調査」と「調査済みだが不明」を区別するため。既存の安禄山・後周恭帝・劉豫の3例が先例）。フィールド自体を省略するのは「まだそのブロックの調査に着手していない」場合のみとする。
- 調査ブロックの完了管理は他グループと同じく `meta.agesCompletedBlocks`（新設）とこのファイルの進捗記録で行う。ブロック内の全員について「データが埋まっている」か「調査済みだが不明」のいずれかの状態になった時点でそのブロックを完了とする。

### 調査方法（トークン効率、2026-07-17確定）

既存213人分の実データの多くは、`deathCause.note`/`accessionRoute.note`（死因・即位経路調査で既に読んだ原典の引用）に「時年◯◯」等の年齢記載がそのまま含まれていたものを転記して得られている。残る151人（2026-07-17時点、明16・前漢14・後漢14・清11など大ブロックが未着手のまま丸ごと残っている）についても、この「既存フィールドの再利用」を起点にした以下の方式で調査する。

1. **既存ノートの再確認を起点にする**：対象者の`deathCause.note`/`accessionRoute.note`/`reigns[].note`/`raw`/`startDateRaw`/`endDateRaw`（すでにJSONに存在し、新規の原典アクセス不要）をエージェントへのプロンプトにそのまま含め、年齢・生年に関する記載がすでに含まれていないか確認させる。
2. **不足分のみ、既知の一次史料箇所を再訪する**：それでも見つからない場合、`deathCause.source.page`/`accessionRoute.source.page`（死因・即位経路調査で既に特定済みの巻・篇）と同じ箇所を対象に、崩御記事・即位記事付近の「時年」「年◯◯」「享年」等の記載のみを確認する（無関係な列伝への探索範囲拡大はしない）。
3. **見つからない場合は「調査済みだが不明」として確定し、無理な深追いはしない**（五胡十六国・十国等の小勢力の君主は原典に生年記録が存在しないことが多く、これ自体が正しい調査結果である）。
4. **ブロック着手順**：まず既存データが丸ごと欠落している大ブロック（明16人・前漢14人・後漢14人・清11人、計55人）を優先する（明・清は実録等により生年月日の記載率が高く収穫が見込める）。その後、五胡十六国・五代十国等の断片的な小勢力（計約96人）を同じブロック区分（グループ2/3と同一の12ブロック）で順次処理する。
5. Workflow運用（並列エージェント・id固定・summary-only return・journal.jsonl経由マージ・ホワイトリスト検証）は既存インフラ（`group2-workflow-token-efficiency`・`merge-validation-automation`・`workflow-id-pinning`）をそのまま流用する。

## 全項目共通の設計判断

- **回数系フィールド（2〜9）はすべて `count` + `events[]` の同型構造**。将来のサイト実装でグラフ化する際に統一的に扱える。
- **`count` は `events.length` と一致させる**。不一致（「少なくともN回」等、正確な件数が確定できない場合）は `count` に判明している最小件数を置き、`note` に不確実性を明記する。
- **0回は明示的に `count: 0, events: []` で記録し、未調査とは区別する**。人物レコードにフィールド自体が存在しない＝その項目が未調査であることを意味する（`meta.status.phases.<key>.status` が `completed` になるまでの間、一部人物のみフィールドを持つ状態が生じる。これは `deathCause` と同じ運用）。
- **各フィールドの `confidence`** は `verification.confidence` と同じ3段階（`high`/`medium`/`low`）。
