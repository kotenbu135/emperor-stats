# 系譜・即位経路グラフスキーマ設計（`data/kinship.json`）

全収録皇帝365人を親子・兄弟・養子・婚姻・即位経路のエッジで結んだ単一グラフ（系譜グラフ）のデータ設計ドキュメント。task.md 項目 6-3（系図）に対応する。2026-07-22 にユーザーと方針確定・フェーズ0（スキーマ設計）として作成。**同日、可視化方式の決定（方式③・縦時間軸）とモック検証を経てスキーマを凍結**（凍結後の変更は原則しない。調査済みデータの再訪問を避けるため、フィールド追加が必要になった場合は保存済み原文パッセージからの再抽出で賄う——「調査運用ルール」節を参照）。

**データは `data/emperors.json` とは別ファイル `data/kinship.json` に置く**（ユーザー決定）。理由：emperors.json の肥大化（既に約7MB）を避ける・並行セッションでの編集衝突面を減らす・グラフ構造（ノード＋エッジ）はレコード配列と設計思想が異なる。皇帝ノードの実体は emperors.json 側にあり、kinship.json からは `id` で参照する（人物情報の重複保持はしない）。

## ゴール像

- 成果物は「1本の木」ではなく**複数の根を持つ1つのグラフ（森・多根DAG＋無向エッジ）**。「どこからでもない建国」（accessionRoute=建国 65人）は根ノードとして存在し、全体が1つの連結成分になることは保証しない。
- 価値は**別王朝間の意外な接続**（関隴集団の姻戚網・王莽⇄前漢・曹⇄漢⇄司馬・蕭氏の斉/梁/後梁など）を面で可視化できること。
- 王朝は実質的建国者（追尊皇帝）から表示できるようにする（例：西晋は司馬懿から。ユーザー要望）。

## スコープルール（ノード収録基準・2026-07-22 ユーザー確定）

ノード＝収録皇帝365人（emperors.json の `id` 参照）＋**ブリッジ人物**（非皇帝、`persons` 配列に実体を持つ）。貴族の系譜は遡れば際限なく繋がるため、ブリッジ人物は以下のいずれかを満たす場合**のみ**収録する：

1. **経路上**: 収録皇帝2人を血縁・婚姻で結ぶ経路上にある人物（例：司馬伷─司馬覲〔西晋⇄東晋元帝の接続鎖〕、独孤信）
2. **一親等（父系）**: 収録皇帝の実父・養父（全員分記録する。祖先をさらに遡るのは、別の収録皇帝または既収録ブリッジ人物に到達する場合のみ）。**実母は基準1・4に該当する（＝別家系との接続に寄与する）場合のみノード化する**（2026-07-22 可視化検討時に確定。全皇帝の生母をノード化すると+200人規模となり「継承＋接続グラフ」という本グラフの性格に合わないため。接続に寄与しない生母の情報は、調査時に保存する原文パッセージ（後述）と succession/kinship エッジの note に残し、ノード・エッジ化はしない）
3. **実在追尊皇帝**: 王朝から皇帝号を追贈され、**正史に事績の記載がある実在人物**（司馬懿・司馬師・司馬昭・曹操・宇文泰・高歓・高澄・楊忠・李虎・李昞・チンギス〜モンケ・ヌルハチ等）。伝説的・儀礼的な遠祖（唐聖祖＝老子、金の函普、清の肇祖〜顕祖、宋の僖祖等）は**ノード化せず** `genealogicalClaims`（系譜主張）に記録する
4. **婚姻当事者**: 別家系の収録ノード同士を結ぶ婚姻の当事者女性（独孤三姉妹・王政君・曹節ら）のみ。皇后の全数収録はしない（同一王朝内で完結する立后は既存 `empressInstallationCount` の領域）

孤立ブリッジ（どのエッジからも参照されない `persons` エントリ）は許容しない（CI でエラー）。

## トップレベル構造

```json
{
  "meta": { ... },
  "persons": [ ... ],
  "edges": [ ... ],
  "genealogicalClaims": [ ... ]
}
```

`meta` は emperors.json と同じ思想で `schemaVersion` / `version`（CalVer・データと共通の CHANGELOG.md で管理）/ `status.overall` / `status.phases` / `completedBlocks` を持つ。フェーズは調査計画（後述）に対応する `succession` / `parentage` / `interdynastic` / `crosscheck` の4つ。

## `persons` — ブリッジ人物ノード

```json
{
  "id": "p-sima-yi",
  "name": "司馬懿",
  "kana": "しばい",
  "aliases": ["晋宣帝"],
  "kind": "追尊皇帝",
  "gender": "male",
  "section": "晋",
  "birthYear": 179,
  "deathYear": 251,
  "yearsApproximate": false,
  "posthumous": { "dynasty": "西晋", "templeName": "高祖", "posthumousName": "宣皇帝" },
  "inclusionReason": ["追尊皇帝", "経路上"],
  "note": "曹魏の重臣。高平陵の変で実権を掌握し西晋の基礎を築いた。孫の司馬炎が受禅した後、宣帝と追尊された。",
  "wikidata": "Q9328",
  "source": { "page": "晋書 巻一 宣帝紀", "lang": "zh-classical" }
}
```

- **`id`**: `p-` プレフィックス＋kebab-case。emperors.json の `id` と衝突しないよう名前空間を分ける（CI で衝突・プレフィックスを検証）。
- **`kana`**: **検索用のひらがな正規化キー**（必須・ひらがなのみ）。site のかな検索は searchKana をひらがなに正規化して照合する方式（`site/src/lib/kana.ts` の `toHiragana` がカタカナをひらがなへ変換）のため、**カタカナ名の人物もひらがなで入れる**（ヌルハチ→`ぬるはち`、チンギス・ハン→`ちんぎすはん`。表示用の読みではない——表示はカタカナ含む `name`/`aliases` を使う）。**調査時に必ず付与する**（後付けは全員再訪問になる。ブリッジ人物の読みはデータ側で完結させ、site 側 `kana-readings.ts` への追記運用は増やさない）。
- **`kind`**: `追尊皇帝` / `宗室` / `外戚` / `后妃・公主` / `その他` の enum。境界ルール（2026-07-22 ブロック1で確定）: `追尊皇帝` は**皇帝号＋廟号の追贈がある**司馬懿・曹操・ヌルハチ型のみ。太上皇の尊号のみ・皇帝制度成立前の先行君主（荘襄王・劉太公型）は `宗室` とする。
- **`gender`**: `male` / `female`。婚姻エッジの表示・検証に使う。
- **`section`**: 表示レーン（必須）。emperors.json の `dynasty.section` と**同じ語彙**を使う（CI で実データの語彙集合への所属を検証。例：`三国` は不正で `三国時代` が正）。可視化のレーン配置・配色に使うため、**「その人物をどの王朝のレーンに表示したいか」で選ぶ**：追尊皇帝は**追尊した王朝の section**（司馬懿は魏の重臣だが `晋`——「西晋は司馬懿から表示」というユーザー要望の実現手段）、それ以外は主たる活動政権の section（独孤信は `北朝`）。
- **`birthYear` / `deathYear`**: 天文年規約の整数（BCE は「歴史年−1 の負数」、emperors.json のイベント年規約と同一。ADDITIONAL_SCHEMA.md 参照）。不明は `null`。可視化はノードを生没年の**中点に固定サイズで配置**するため（可視化節参照）厳密値は必須でない——概数しか得られない場合は推定値を入れ `yearsApproximate: true` を付す（表示側で「生没年推定」と明示）。両方 `null` の場合は表示側が隣接ノードから配置年を推定する。
- **`posthumous`**: 追尊皇帝のみ。どの王朝が何と追尊したか。
- **`inclusionReason`**: スコープルール1〜4に対応する `経路上` / `一親等` / `追尊皇帝` / `婚姻当事者` の配列（複数該当可）。収録判断の監査可能性のために必須。
- **`wikidata`**: QID。**調査時に取得を試みる**（英語版〔task.md 項目5〕の nameEn 取得・肖像照合・突合フェーズで使う。見つからない場合は `null` 可）。
- **`note` / `source`**: 既存データと同じ運用。source は原典（正史の紀・伝・表）必須。

## `edges` — エッジ（3種）

全エッジ共通のフィールド：

```json
{
  "type": "succession | kinship | marriage",
  "from": "ノードid（皇帝id または p-*）",
  "to": "ノードid",
  "veracity": "verified",
  "confidence": "high",
  "note": "自然文での説明",
  "source": { "page": "...", "lang": "zh-classical" }
}
```

- **`veracity`**（真偽区分・全エッジ必須）:
  - `verified` — 正史の紀・伝・表で裏付けられる関係
  - `claimed` — 当事者・王朝側の主張であり史実として確認できない関係（例：劉備の漢室後裔、劉淵の「漢の外甥」論理による劉姓）。主張の出所を `note` に明記
  - `disputed` — 複数の原典・学術的通説が対立する関係（例：秦始皇帝の実父呂不韋説は『史記』内でも矛盾があり disputed）
- **`confidence`**: `high` / `medium` / `low`（既存運用と同じ。veracity とは独立の軸——「claimed であることが high confidence」はあり得る）。
- **`source`**: 原典必須。Wikipedia を根拠にしない（既存の禁止出典ルールを CI で同様に適用）。

### 1. `succession` — 継承エッジ（有向: 先代 → 当人）

「誰から皇位を得たか」。`to` は必ず収録皇帝。

```json
{
  "type": "succession",
  "from": "tang-gaozu",
  "to": "tang-taizong",
  "category": "内禅",
  "relationToPredecessor": "子",
  "isRestoration": false,
  "veracity": "verified",
  "confidence": "high",
  "note": "玄武門の変の後、高祖から譲位を受けた。",
  "source": { "page": "旧唐書 巻一", "lang": "zh-classical" }
}
```

- **`category`**: ADDITIONAL_SCHEMA.md の accessionRoute 9分類（世襲/簒奪/禅譲/内禅/擁立/復位/建国/不詳/諸説あり）と同じ enum。
- **`relationToPredecessor`**: 先代との続柄（必須・enum）。**「新帝は先代の◯◯」の向きで記す**（太宗は高祖の`子`、武則天は睿宗の`母`、楊堅は静帝の`外祖父`）。語彙：`子` / `養子` / `孫` / `曾孫` / `弟` / `兄` / `甥` / `姪` / `叔父` / `伯父` / `従兄弟` / `同族（遠縁）` / `父` / `母` / `祖父` / `外祖父` / `女婿` / `舅（妻の父）` / `外戚（その他）` / `無血縁` / `不明` / `その他`（`その他` は note で説明必須）。
  - 目的：(1)「世襲134名の続柄内訳」等の統計を parentage フェーズ完了を待たず出せる (2) エッジ表示ラベル (3) kinship グラフから導出した続柄との**相互検証**（両フェーズ完了後に CI で突合し、矛盾＝どちらかの調査ミスを機械検出する）。即位記事に続柄はほぼ必ず書かれており、succession 調査時の同時取得はコストゼロに近い。
- **本数の規約**:
  - 各皇帝は**初回即位に対応する主エッジを最大1本**持つ（`isRestoration: false`）。ただしこの一意性は `veracity` が `verified` / `claimed` のエッジに適用し、**`disputed` の主エッジは対立説の併記として複数許容する**（例：先代が誰かについて原典が対立する場合。accessionRoute=`諸説あり` の皇帝は disputed エッジのみでもよい）。
  - `建国`（先行政権からの継承なし）の皇帝は主エッジを持たない＝グラフの根（例：漢高祖・明太祖）。ただし先行君主から君主位を実質的に継承したうえで称帝した建国（例：西夏景宗李元昊＝父・李徳明から西平王位を継承し1038年に称帝〔accessionRoute=建国〕）は、`from` に該当ノード（追尊帝ブリッジ含む）を置いてよい。なお「先行君主からの継承＋from あり」でも `category` はあくまで emperors.json の accessionRoute に従う——ホンタイジ（太祖崩御後に諸貝勒の推戴で後金カンを継承→後に称帝）は accessionRoute=`擁立` なので、from=ヌルハチ・category=`擁立` のエッジになる（category を勝手に `建国` へ読み替えない。CI が不一致を検出する）。
  - **復位**は追加エッジ（`isRestoration: true`、`category: "復位"`、`from` は復位時に皇位を得た相手。例：唐中宗の第2次即位は from 武則天）。復位皇帝の主エッジの `category` には初回即位の経路を入れる（accessionRoute.note に記録済みの判定を構造化する）。
- **CI 整合**: `isRestoration: false` かつ非 disputed のエッジの `category` は emperors.json の `accessionRoute.category` と一致しなければならない（復位皇帝〔accessionRoute=復位〕は除く。この場合は逆に `isRestoration: true` エッジの存在を要求する）。
- **並立政権**: 継承エッジは「その政権内で誰から直接皇位を得たか」で判定する。政権を跨ぐ接続は血縁・婚姻エッジに担わせる。

### 2. `kinship` — 血縁エッジ（有向: 親 → 子。兄弟のみ無向扱い）

```json
{
  "type": "kinship",
  "relation": "実父",
  "from": "p-sima-zhao",
  "to": "jin-wudi",
  "childOrder": 1,
  "veracity": "verified",
  "confidence": "high",
  "note": "",
  "source": { "page": "晋書 巻三 武帝紀", "lang": "zh-classical" }
}
```

- **`relation`**: `実父` / `実母` / `養父` / `養母` / `兄弟姉妹` の enum。
  - **養子はフラグではなく独立 relation**（五代の義児継承〔李嗣源・李従珂・柴栄〕、南宋孝宗などで本質的）。実親も判明していれば実父/実母エッジを併記してよい（例：柴栄は郭威の養子かつ柴守礼の実子）。
  - **`兄弟姉妹` は導出可能なら張らない**: 両者の父（または母）エッジが記録されていれば兄弟関係はグラフから導出できるため、明示エッジは張らない。親が不明・収録基準外だが兄弟関係が原典に明記されている場合のみ使う（無向として扱い、`from`/`to` の順序に意味はない）。
- **`childOrder`**（排行・実父/養父エッジの任意フィールド）: 原典の「第n子」表記の n（整数・1始まり。例：「高祖第二子」→ 2）。系図の兄弟並び順に使う（生年不明の皇子が多く生年ソートでは代替できない）。本紀冒頭の出自記事にほぼ必ずあるため**調査時に同時取得する**（後付けは全員再訪問）。原典に排行の記載がない場合は省略。
- **`primaryLineage`**（レイアウト主系統・任意フィールド）: 実父と養父の両エッジが並存する場合に、系図ツリーでどちらの下にぶら下げるかを指定する（`true` を付けたエッジが主系統・子ごとに最大1本）。**デフォルト規則は「succession エッジの from と同一人物の親エッジ＞養父＞実父」**（柴栄は郭威〔養父・継承元〕が主系統）で、規則どおりなら省略してよい。規則と異なる表示にしたい場合のみ明示する。
- 収録皇帝の**実父・養父は全員分調査・記録する**（不明の場合も「調査済みだが不明」を確定させる。グループ5〔ages〕と同じ運用。不明はエッジ不存在＋当該皇帝ブロックの調査ノートで表現）。実母はスコープルール2のとおり接続に寄与する場合のみエッジ化し、それ以外は調査ノート・note に残す。

### 3. `marriage` — 婚姻エッジ（無向）

```json
{
  "type": "marriage",
  "from": "p-dugu-qieluo",
  "to": "sui-wendi",
  "veracity": "verified",
  "confidence": "high",
  "note": "独孤信の七女・文献皇后独孤伽羅。姉は北周明帝の后、四姉は唐高祖の母＝北周・隋・唐を結ぶ姻戚網の要。",
  "source": { "page": "隋書 巻三十六 后妃伝", "lang": "zh-classical" }
}
```

- 無向。CI では正規化のため `(from, to)` を辞書順に持つことを推奨（重複検出用）。
- **別家系の収録ノード同士を結ぶ婚姻のみ記録**する（スコープルール4）。同一王朝内で完結する立后は対象外。

## `genealogicalClaims` — 系譜主張（ノード化しない遠祖）

伝説的・儀礼的な祖先の主張はエッジではなくここに記録する（スコープルール3の受け皿）：

```json
{
  "claimant": "tang-gaozu",
  "claimedAncestry": "老子（李耳）・西涼武昭王李暠を経る隴西李氏",
  "note": "唐王朝は老子を聖祖として太上玄元皇帝に追尊した。隴西李氏系譜には唐代から疑義があり、陳寅恪以来の研究では趙郡李氏傍系または非漢族系の可能性が指摘される。",
  "source": { "page": "新唐書 巻一 高祖紀・宗室世系表", "lang": "zh-classical" }
}
```

- **`claimant`**: 主張の主体となる王朝の建国者（または当該皇帝）の `id`。
- 例：劉備の漢室後裔（中山靖王之後）、蕭斉・蕭梁の蕭何後裔、金の函普、清の肇祖〜顕祖、宋の僖祖〜翼祖。

## 情報源・調査方針

[DEATH_CAUSE_SCHEMA.md](DEATH_CAUSE_SCHEMA.md)・[docs/process/RESEARCH_PROCESS.md](../../docs/process/RESEARCH_PROCESS.md) と同じ方針に従う。原典（正史）を第一次情報源とし、WebSearch の要約のみを根拠に確定しない。**スクリプトによる自動生成は禁止**（既存 note・在位順からの機械推定は「確認用ドラフト」までとし、エッジ1本ごとに原典で個別確認する）。

- **一次情報源**: 本紀冒頭の出自記事・宗室伝・外戚伝・后妃伝、および系譜「表」（遼史皇族表・金史宗室表・元史宗室世系表・明史諸王世表・清史稿皇子世表など）。
- **系譜表の所在（2026-07-22 下見）**: 表は china-history には無く **daizhigev20 側にのみ存在**（遼史皇族表・金史宗室表・明史諸王世表・元史/宋史の宗室世系表を確認済み）。新唐書は「世系」ヒットが少なく宗室世系表の完全収録が未確認——**着手時に書ごとに実在・可読性を必ず確認**する（CORPUS_NOTES.md の運用）。
- **Wikidata**: P22（父）/P25（母）/P26（配偶者）/P1365（先代）を**漏れ検出スクリーニングにのみ**使う（情報源にしない・CI 非組込・bulk API は小規模検証から。既存 QA 運用と同じ）。QID は365人全員に付与済み。

### 調査運用ルール（スキーマ凍結にともなう再調査防止策・2026-07-22 確定）

1. **原文パッセージの保存**: 調査時に読んだ即位記事・出自記事の原文（該当箇所の抜粋）を人物ごとに `_corpus_cache/` の延長として保存する。スキーマ凍結後にフィールド追加が必要になった場合、**原典の「再読」でなく保存パッセージからの「再抽出」**で賄うための保険（未知の未知への唯一の実効策）。
2. **王朝間エッジはクラスタ単位で初回接触時に取り切る**: 王朝ブロック単位の調査中に王朝を跨ぐ縁戚（例：北周調査中の独孤信の娘たち→隋・唐への接続）に当たったら、**その場で跨ぎ先も含めてクラスタ全体を記録**する。ブロック順の都合で後回しにすると同じ史料を二度読みすることになる。
3. **調査時の同時取得が必須の項目**（後付けは全員再訪問になるもの）: `relationToPredecessor`（即位記事）・`childOrder`（出自記事）・persons の `kana` / `section` / `wikidata` / 生没年（推定なら `yearsApproximate`）。

## CI 検証（`scripts/validate_kinship.py`）

kinship.json は emperors.json を参照するため専用スクリプトで検証し、`.github/workflows/validate-data.yml` に組み込む。チェック内容：

**構造（常時・エラー）**:
- persons: `id` 一意・`p-` プレフィックス・emperors.json の id と非衝突・enum（kind/gender/inclusionReason）・`kana` 必須（ひらがな）・`section` 必須（emperors.json の `dynasty.section` 語彙集合に所属）・source 必須
- edges: `from`/`to` が実在ノード（皇帝 id または persons）・enum（type/relation/category/relationToPredecessor/veracity/confidence）・source 必須・重複エッジなし（marriage は無向正規化して判定）
- succession: `to` は皇帝・`relationToPredecessor` 必須（`その他` は note 必須）・主エッジ（isRestoration=false・veracity≠disputed）は皇帝ごとに最大1本・非 disputed 主エッジの category が emperors.json の accessionRoute.category と整合（復位皇帝の例外規約は上記）・自己ループなし
- kinship: 実父/養父の from は male、実母/養母の from は female（gender 判明時のみ）・親子の循環なし（DAG 検証）・verified の実父エッジは子ごとに最大1本（disputed 併存は可）・`childOrder` は1以上の整数・`primaryLineage: true` は子ごとに最大1本
- 孤立ブリッジ（被参照ゼロの persons）なし
- genealogicalClaims: `claimant` の実在・source 必須
- 禁止出典（Wikipedia 等。validate_emperors.py と同一基準）

**網羅性（フェーズ completed 後に有効化・エラー）**:
- `succession` 完了後: 全皇帝が主エッジを持つ（disputed のみでも可）、または accessionRoute=建国/不詳/諸説あり
- `parentage` 完了後: 全皇帝が実父または養父エッジを持つ、または当該ブロックの調査ノートで「不明確定」
- 両フェーズ完了後: `relationToPredecessor` と kinship グラフから導出した続柄の突合（矛盾＝どちらかの調査ミスの機械検出。導出不能な続柄〔経路が noded されない遠縁など〕は対象外）

## 可視化の決定事項（2026-07-22 ユーザー決定・モック検証済み）

サイト側実装はデータ完了後だが、**スキーマの必須フィールドは可視化方式から逆算して確定した**ため、決定内容をここに記録する（サイト実装時は docs/site-design/LAYOUT.md にも記録を残すこと）。

- **方式③「全体1画面のインタラクティブグラフ」＋縦軸=時間（上→下に時代が下る）**。複数候補（時間軸大河型・王朝単位世代ツリー型・全体グラフ型）から捏造サンプルデータのモック（ハードケース: 呂不韋 disputed 実父・柴栄の実父/養父・武則天の皇帝間婚姻と母ノード・独孤姉妹の王朝間婚姻網・隋恭帝の並立在位・中宗/睿宗の複数在位・建国ルート・唐高祖の系譜主張、を含む）で検証して確定。
- **視覚エンコーディング**: 皇帝=在位期間の長さを持つカプセル（色=王朝、レーン=時代区分）／ブリッジ人物=固定サイズの破線枠ノード（生没年の**中点**に配置。生没年スパンで描くと巨大化して皇帝を食うため不採用）／継承=アクセント色の矢印＋カテゴリラベル／血縁=実線／養子=破線／婚姻=二重線（無向）／disputed・系譜主張=点線＋?マーク／複数在位=カプセル複数＋同一人物点線コネクタ／建国=根バッジ。
- **モックで判明した本番実装の必須処理**: 短在位カプセルへの最小高付与＋同一レーン内の衝突押し下げ（時間軸は**近似線形**になる）／短カプセルのラベル左外置き・継承ラベル右外置き／レイアウトはビルド時計算（365+α ノードのエッジルーティング・ラベル衝突回避を含む）。**→ 押し下げ方式は実データ検証（下記）で「年目盛りとのズレ」のユーザー指摘を受けて廃止し、密集期間だけ軸を局所的に引き伸ばす単調写像（ノードと目盛りが常に一致）へ変更した**（実装詳細は docs/site-design/LAYOUT.md）。
- **本番実装時の設計項目（バックログ）**: ズーム/パン＋王朝ジャンプ or ミニマップ（縦2,200px 超になるため必須級）／2ノード選択の接続経路ハイライト・先祖子孫の系統ハイライト／継承カテゴリ別フィルタ／かな検索・個別ページ導線・ツールチップ肖像画／SEO・SSR テキスト（継承チェーンのテキスト版。クロール可能リンクは LazyMount 外の既存原則）／テーブルビュー／モバイル方針（横スクロール＋ピンチズーム）。
- **実データでの描画検証（2026-07-22・フェーズ1ブロック1・2完了時点）**: 調査済み36人・継承エッジ29本を試作ページ `/kinship`（noindex・ナビ非登録。実装記録は docs/site-design/LAYOUT.md）で実際に描画し、**スキーマは現状のまま通用する（変更不要で調査続行可）**ことを確認した。行使できた機能: succession 3カテゴリ・veracity disputed・relationToPredecessor・ブリッジ人物・yearsApproximate・section=レーン・建国/擁立の根。データ側の留意点2つ: ①**劉永の section「梁」は南朝梁と語彙を共有**するため、サイト側のレーン割当は section 単独では決まらず id 個別指定が要る（スキーマ違反ではない。以後も同名 section が出たら同様に処理）②孺子嬰→王莽のように**ブリッジ人物（生没中点配置）が to より下に来る継承エッジ**が正当に存在する——描画側の側面アタッチで解決済みで、データ側の対応は不要 ③カプセル=皇帝在位・ブリッジ=生没中点のため、**継承エッジの視覚的な長さは継承イベントの時期と一致しない**（荘襄王→始皇帝の世襲は前247年の秦王位継承だが、エッジは前221年開始のカプセル上辺まで伸びる）。追尊帝・先行君主→帝号以前の在位を持つ皇帝（曹操→曹丕・司馬氏など）で今後も繰り返し現れる型なので、本実装時にアンカー位置か注記の方針を決める（正しさの問題ではなく読み取りの問題）。
